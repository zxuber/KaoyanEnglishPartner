# 工程目录结构与搭建规范

> 用途：以后从 0 到 1 搭建同类系统时，把本文档提供给新会话/新开发者，即可按当前工程的风格生成前后端项目结构、技术栈和基础工程约定。本文只总结工程设计，不包含现有项目业务源码。

## 1. 整体工程形态

当前工作区采用前后端分离的双项目结构：

```text
workspace/
├── xxx_web/          # 前端项目，Vue 2 管理后台/工具平台
└── xxx_backend/      # 后端项目，Spring Boot + Maven 服务
```

整体风格适合构建内部管理后台、联调工具平台、配置平台、数据查询平台、流程编排工具等系统。

核心设计原则：

- 前端负责页面、菜单、权限跳转、表单交互、JSON/代码编辑器、管理后台组件化封装。
- 后端负责 REST API、登录鉴权、菜单权限、业务服务、数据库访问、多数据源、外部系统调用、异步任务、文件处理等。
- 前端统一通过固定 API 前缀访问后端，例如 `/jointApi`，由本地开发代理或 Nginx 转发到后端服务上下文。
- 后端统一设置服务上下文路径，例如 `/joint`，对外 API 和健康检查都挂在该上下文下。
- 页面路由尽量与 `views` 目录路径保持一致，便于根据菜单配置动态加载页面组件。
- 所有环境差异（OS、域名、端口、外部命令、账号密码）统一收口到配置层，不写死在业务代码中。

## 1.1 环境配置原则

无论前端还是后端，都应遵循：

- 业务代码不直接写死 `localhost`、盘符路径、`python.exe` 等平台细节
- 前端通过 `.env.*` 或统一运行时配置模块管理 API 基地址
- 后端通过环境变量或 profile 管理数据库、外部命令、第三方服务配置
- Windows / macOS / Linux 的差异只体现在配置值，不体现在业务逻辑
- 真实 API Key、Secret、Token 不进入 Git 历史；仓库只保留环境变量入口和示例值

## 4. 前端设计思路

### 4.1 应用入口

`src/main.js` 只负责组装应用，不写业务逻辑：

- 引入 Vue、根组件、路由、Vuex。
- 注册 Element UI、内部组件库、JSON 编辑器、JSON 查看器等插件。
- 注册全局 HTTP 实例、API 集合、时间工具等。
- 挂载到 `#app`。

推荐约定：

- 全局请求对象挂到 `Vue.prototype.$http`。
- API 方法集合挂到 `Vue.prototype.$api`。
- 时间工具挂到 `Vue.prototype.$moment`。
- 全局样式和 UI 框架样式在入口统一引入。

### 4.2 HTTP 与 API 封装

请求分两层：

```text
common/http.js  # Axios 基础实例：baseURL、headers、token、错误处理、响应解包
common/api.js   # 按后端接口封装业务方法
```

设计要求：

- 页面不要直接写 Axios 请求地址，应调用 `api.js` 中的方法。
- Token 统一从本地存储读取并通过请求拦截器带上。
- 响应错误、登录失效、后端异常提示统一在 `http.js` 处理。
- 如存在独立服务，例如 SSE、聊天、AI 接口，可单独拆 `chatApi.js`，不要混在主业务 API 中。

### 4.3 路由与菜单

路由采用 `hash` 模式，适合静态资源部署在任意路径下，减少 Nginx rewrite 配置。

推荐路由策略：

- 固定路由只放登录页、错误页、少量无需菜单权限的页面。
- 登录后调用后端权限接口获取用户、菜单、环境、默认首页等信息。
- 根据后端返回菜单动态注册路由。
- 菜单路径与 `src/views` 文件路径保持一致，通过路径约定自动加载组件。

示例约定：

```text
菜单 path: /manage/projectList
页面文件: src/views/manage/ProjectList.vue
```

路径转组件规则：

- 取 path 最后一段首字母大写作为 `.vue` 文件名。
- 前面的路径层级与 `views` 下目录保持一致。
- 页面组件如不希望出现在菜单，可通过组件元信息隐藏。

权限守卫逻辑：

- `/login`、`/500` 等特殊页面直接放行。
- 无 token 跳转登录页。
- 有 token 但未加载用户信息时，先拉取权限菜单并动态注册路由。
- 权限接口失败可重试，超过次数进入系统异常页。
- 无匹配路由时进入后端指定的缺省页面或无权限页。

### 4.4 Vuex 状态

Vuex 用于保存全局共享状态，不承载复杂业务流程。

推荐状态：

```text
user                       # 当前登录用户
menus                      # 当前用户菜单
envs                       # 可选环境列表
project / currentProject   # 当前项目/业务上下文
xxxOptions                 # 下拉选项缓存
xxxMap                     # 字典、机构、产品等映射缓存
```

推荐写法：

- `mutations` 只做同步赋值。
- `actions` 负责懒加载全局下拉选项、字典、机构映射等。
- 对不会频繁变化的选项数据做前端缓存，避免页面间重复请求。

### 4.5 页面与组件分层

页面级组件放 `views`，可复用组件放 `components`。

拆分标准：

- 只服务某个页面的内容，放在对应页面或页面模块目录内。
- 跨多个页面复用的表单项、选择器、编辑器、树、头部、返回按钮等，放 `components`。
- 工具函数和校验逻辑放 `utils`，不要塞进页面。
- 全局通用方法放 `common/common.js`，但不要让它变成业务杂物箱。

页面模块命名建议：

```text
views/manage/     # 项目、用户、文档、Mock、数据模型等管理页面
views/config/     # 配置页面
views/db/         # 数据源、同步、SQL/数据相关页面
views/tool/       # 缓存、文件、转换、调试工具页面
views/err/        # 错误页
views/chat/       # 对话或智能助手页面
```

### 4.6 构建与部署

`vue.config.js` 推荐配置：

- `productionSourceMap: false`，生产包不输出源码映射。
- `publicPath: "./"`，静态资源使用相对路径，便于部署到任意目录。
- `devServer.port` 使用固定本地端口。
- `devServer.proxy` 按 API 前缀代理到后端，例如 `/jointApi`。
- 如存在 SSE 流式接口，开发代理和 Nginx 都要关闭缓冲或延长超时。

部署方式：

- 构建产物输出到 `dist`。
- CI 执行 `npm install` 和 `npm run build`。
- 构建后把 `deploy` 目录复制进产物，方便部署机执行 Nginx 脚本。
- Nginx 提供静态资源访问，同时反向代理后端 API 和其他服务。

Nginx 推荐约定：

```text
/                  -> 前端 dist
/jointApi/         -> 后端 /joint/
/health            -> 后端 actuator health
/sse/...           -> 流式服务，关闭 proxy buffering
/chat/...          -> 独立聊天/AI 服务
```

## 7. 后端设计思路

### 7.1 分层职责

推荐请求链路：

```text
Controller -> Service -> DAO/Mapper -> Database
                 |
                 -> Remote
                 -> Handler / Processor
                 -> Helper / Tool / Util
```

职责边界：

- `controller`：只处理 HTTP 入参、权限入口、调用服务、返回响应，不写复杂业务。
- `service`：组织业务流程、事务边界、调用 DAO/remote/handler。
- `dao`：封装多个 mapper 的组合查询或复杂数据访问。
- `mapper`：MyBatis 数据访问接口，与 XML 一一对应。
- `pojo/req`：请求对象。
- `pojo/res`：响应对象。
- `pojo/vo`：前端展示对象。
- `pojo/po`：数据库持久化对象。
- `pojo/domain`：领域模型。
- `pojo/data`：流程中间数据、上下文数据。
- `remote`：外部服务调用隔离层，不让 HTTP/RPC 细节散落在 service。
- `handler`：可插拔业务处理节点，适合策略、责任链、按类型分发。
- `processor`：较长业务流程的编排器，适合发送、接收、构建、校验等流程节点。
- `config`：集中放 Spring Bean、缓存、多数据源、JSON、调度等配置。
- `aop`：全局日志、异常处理、动态数据源切换、上下文注入等横切逻辑。

### 7.2 多数据源与 Mapper 组织

后端适合多数据源场景。推荐规则：

- Java mapper 接口按数据源或业务库拆包，例如 `mapper/local`、`mapper/biz`、`mapper/transaction`。
- XML mapper 在 `resources/mapper` 下使用同名目录对应。
- 数据库实体 PO 放在 `pojo/po/{datasource}` 下，与 mapper 目录保持一致。
- 动态数据源切换通过注解、AOP 或 dynamic-datasource 统一处理，不在业务代码里手动拼连接。
- 本系统自有表优先放 `local` 目录，外部业务库按实际库名或业务域建目录。

示例：

```text
java/.../mapper/local/UserMapper.java
resources/mapper/local/UserMapper.xml
java/.../pojo/po/local/UserPO.java
```

### 7.3 配置与运行

基础配置建议：

```properties
server.port=8899
server.servlet.context-path=/apiContext

apollo.bootstrap.enabled=true
apollo.bootstrap.eagerLoad.enabled=true
app.id=xxx-backend

spring.servlet.multipart.max-file-size=10MB
spring.servlet.multipart.max-request-size=10MB
```

推荐约定：

- 本地和部署环境端口固定，便于前端代理和 Nginx 配置。
- 服务统一配置 context-path，例如 `/joint`、`/admin`、`/tool`。
- 配置中心使用 Apollo，敏感配置和多环境配置不要写死在代码里。
- 开启 Actuator，提供 `/actuator/health` 给 Nginx 或部署平台探活。
- 日志使用 `logback-spring.xml` 管理。

### 7.4 安全与权限

推荐使用 Spring Security + JWT：

- 登录成功后返回 token。
- 前端把 token 存储在本地并随请求发送。
- 后端过滤器解析 token 并注入登录上下文。
- 权限菜单由后端返回，前端根据菜单动态注册路由。
- 用户、角色、菜单、项目上下文等信息统一由后端权限接口返回。

### 7.5 异常、日志与响应

推荐统一处理：

- `GlobalExceptionHandler` 处理业务异常、参数异常、系统异常。
- `GlobalLogHandler` 或 AOP 记录请求日志、耗时、核心上下文。
- 自定义 `ErrorCode` / `ErrorCodeEnum` 管理错误码。
- Controller 返回统一响应结构，避免每个接口格式不一致。

### 7.6 可扩展业务流程

当业务存在多类型、多场景、多节点流程时，优先使用：

- 注解标识 handler 类型。
- Factory 根据类型选择 handler。
- ChainHandler 或 Processor 串联多个处理步骤。
- StrategyContext 管理策略实现。

适用场景：

- 不同业务类型走不同处理逻辑。
- 不同机构/渠道/场景需要不同实现。
- 请求构建、发送、接收、解析、校验、落库等流程可拆节点。

## 8. 前后端接口协作约定

推荐路径映射：

```text
前端本地请求: /jointApi/xxx
本地代理目标: http://backend-host:8899/joint/xxx
生产 Nginx:  /jointApi/ -> http://backend-host:8899/joint/
后端服务:    server.servlet.context-path=/joint
```

推荐接口分类：

- `/auth`、`/user`：登录、用户、菜单、权限。
- `/common`、`/options`：下拉选项、字典、环境。
- `/manage`：管理类接口。
- `/config`：配置类接口。
- `/tool`：工具类接口。
- `/db`：数据源、同步、查询类接口。
- `/mock`：Mock 数据、模拟请求。
- `/file`：上传下载。

前端新增页面时的标准流程：

1. 后端新增 Controller 接口。
2. 前端在 `common/api.js` 增加 API 方法。
3. 前端在 `views/{module}` 新增页面。
4. 后端菜单配置增加页面 path。
5. 前端登录后通过动态菜单自动注册路由。

## 9. 从 0 到 1 搭建提示词

以后新开会话搭建系统时，可直接使用下面这段提示词：

```text
请按以下工程规范从 0 到 1 搭建一个前后端分离系统：
```

## 10. 新项目命名建议

如果新系统名为 `demo`，推荐：

```text
demo_web
demo_backend
```

前端：

```text
name: demo-web
dev port: 9090
api prefix: /demoApi
```

后端：

```text
artifactId: demo-backend
server.port: 8899
server.servlet.context-path: /demo
app.id: demo-backend
```

Nginx：

```text
/demoApi/ -> http://backend-host:8899/demo/
/health   -> http://backend-host:8899/demo/actuator/health
```
