## 一、产品概述

### 1.1 产品定位

一款基于大语言模型的考研英语一/二智能陪练小程序。核心逻辑是“先诊断、后定标、再分阶段执行”，为每个用户生成专属学习方案，并在执行过程中通过 AI 对话式交互完成单词速记、阅读精读、作文批改等核心训练动作。

### 1.2 目标用户

- 英语基础薄弱（四级未过/低分飘过），对考研英语感到无从下手的考生
- 其他科目复习压力大，每天只能投入 1.5-2 小时给英语的考生
- 需要“被带着走”而非“自己规划”的自驱力一般型考生

### 1.3 核心价值

| 用户痛点 | 产品解法 |
|----------|----------|
| 不知道自己的水平能考多少分 | 入门诊断 + LLM 生成专属分数拆解 |
| 背单词坚持不下去，背了就忘 | 对话式闪卡 + 2 秒计时 + 标记->复现循环 |
| 做阅读读不懂，不知道错在哪 | 逐句精读引导 + 错因分类标注 + 误解本沉淀 |
| 作文不会写，背范文套不上 | 模板工厂 + LLM 逐句批改 + 个人模板库 |
| 不知道自己有没有进步 | 数据看板：词汇掌握率、阅读正确率曲线、预计分数 |

### 1.4 与大模型的结合点总览

| 模块 | LLM 调用场景 | 优化策略 |
|------|-------------|----------|
| 入门诊断 | 根据用户 7 项基础信息生成专属学习计划、分数拆解、阶段节奏 | 硬约束规则表嵌入 prompt，输出前校验总分偏差 <= 3 分 |
| 单词速记 | 语义判断用户口述释义是否正确（非精确匹配） | **四级本地检查**：直接包含 → 关键词匹配 → 近义词映射 → 编辑距离；本地全挂且零字重叠才 fallback LLM；LLM 具备单词级降级 + 用户级熔断 |
| 阅读精读 | 长难句智能拆解、段落主旨总结、错题诊断（单词/语法/逻辑三维归因） | 相同真题篇目的拆解/解析结果 Redis 缓存，同篇二次请求直接返回 |
| 作文训练 | 模板智能生成、用户作文逐句批改、改写润色建议 | 结构分用规则引擎（段落完整性/字数/语法错误数），内容分用 LLM |
| 翻译练习 | 用户译文与参考译文语义对比打分 | 简单句用向量相似度，复杂句用 LLM |
| 进度教练 | 根据用户当前数据生成阶段性鼓励文案与节奏微调建议 | 里程碑文案预置模板 + LLM 仅填充动态变量（数字/题型名称） |

**模型选型：DeepSeek-V3**
- API 兼容 OpenAI 格式，切换成本低
- 中文理解能力满足考研辅导场景
- 成本可控，高频调用场景（单词判断 fallback、长难句拆解）扛得住

**调用原则：**
- 能本地判定的绝不调 LLM（单词判断 90% 走本地）
- 能缓存复用的绝不重复调（阅读解析按篇目缓存）
- 必须调 LLM 的任务设置 temperature=0 降低随机性
- LLM fallback 具备自动熔断机制：同词触发 LLM 超过阈值则降级为"跳过校验"；同用户连续 LLM false 超过阈值则当日暂停 fallback

### 3.1 入门诊断与计划生成

#### 3.1.1 基础信息采集（7 题问卷）

用户首次进入小程序，逐题展示（一屏一题，减少跳出）：

| 题号 | 问题 | 题型 | 选项/输入 |
|------|------|------|-----------|
| Q1 | 你考英语一还是英语二？ | 单选 | 英语一 / 英语二 |
| Q2 | 目标分数是多少？ | 滑块 | 40-80 分，默认 60 |
| Q3 | 距离考试还有多少天？ | 日期选择器 | 自动计算距 12 月考研的天数，也可手动输入 |
| Q4 | 每天能稳定投入多长时间给英语？ | 单选 | 1h / 1.5h / 2h / 2.5h / 3h+ |
| Q5 | 周末是否有时间学习？ | 单选 | 有时间 / 没时间 / 不确定 |
| Q6 | 你目前的英语水平？ | 单选 | 四级未过 / 四级低分飘过 / 四级高分 / 六级已过 / 六级高分 |
| Q7 | 手头已有的资料？（多选） | 多选 | 红宝书 / 恋练有词 / 刘晓燕系列 / 考研真相 / 张剑黄皮书 / 王江涛作文 / 还没有资料 / 其他 |

#### 3.1.2 LLM 生成专属学习方案

采集完成后，将 Q1-Q7 的答案组装为 prompt，调用大模型生成以下内容：

**方案输出结构（固定格式）：**

\`\`\`
用户画像：
- 英语基础：{Q6答案}
- 目标：英语{Q1答案} {Q2答案}分
- 可用时间：{Q4答案}/天，周末{Q5答案}
- 剩余天数：{Q3答案}天
- 已有资料：{Q7答案}

=== 分数拆解 ===
| 题型 | 满分 | 目标分 | 难度评级 |
|------|------|--------|----------|
| 阅读理解 | 40 | {动态计算} | ★★★★★ |
| 大作文 | 20 | {动态计算} | ★★★ |
| 小作文 | 10 | {动态计算} | ★★ |
| 新题型 | 10 | {动态计算} | ★★ |
| 翻译 | 10 | {动态计算} | ★ |
| 完形填空 | 10 | {动态计算} | ☆ |
合计：{动态计算} 分

=== 复习节奏 ===
第一阶段（第 1-X 天）：基础清零——词汇+语法长难句
第二阶段（第 X-Y 天）：阅读攻坚——真题精读+错题归因

{根据用户可用时间动态分配}

\`\`\`

**LLM 调用说明：**
- 分数拆解逻辑：根据用户基础和目标分动态分配，例如基础弱的用户在阅读和作文上分配更高权重，完形分配更低
- 阶段天数：根据剩余总天数和每日可用时间自动计算
- 输出需为结构化数据（JSON），前端渲染为卡片

**交互细节：**
- 方案生成后展示“专属学习方案”卡片页，支持一键保存为首页置顶
- 提供“调整目标”入口，修改 Q2 后重新生成
- 每个阶段开始时，首页自动切换为该阶段的每日任务清单

### 3.2 单词速记模块（第一阶段核心）

#### 3.2.1 词库设计

- 内置《红宝书》必考词（约 26 Unit，约 1800 词）、基础词（约 1800 词）、超纲词
- 每个单词数据结构：{英文, 音标, 中文释义, 词性, 所属Unit, 真题例句, 掌握状态, 标记次数, 最后出现时间}

#### 3.2.2 对话式闪卡——交互流程

单词卡片出现在屏幕中央，展示英文单词 + 音标，中文释义被遮罩隐藏。同时启动 2 秒倒计时，进度条从绿色->黄色->红色。用户按住麦克风按钮说出中文释义，松手后语音识别为文本。

**具体规则：**

1. **单词卡片出现**：屏幕中央展示英文单词 + 音标，中文释义被遮罩隐藏
2. **2 秒倒计时**：卡片出现同时启动 2 秒倒计时，进度条从绿色->黄色->红色
3. **语音输入**：用户按住麦克风按钮说出中文释义，松手后语音识别为文本
4. **逐级判定**：详见 3.2.5 单词判定策略。核心流程——先走本地四级检查（直接包含→关键词→近义词→编辑距离），全挂且与标准释义零字重叠时才调 LLM 做语义判断；LLM 调用有自动熔断保护。
5. **倒计时内正确** -> 绿色打勾，进入下一个词
6. **倒计时内未说出或错误** -> 红色标记，该词加入“标记词池”
7. **用户主动点“不认识”** -> 同样加入标记词池，并展示中文释义 3 秒后再进入下一个
8. **每轮 20 个词**：一轮结束后展示本轮正确率（如 15/20）

#### 3.2.3 标记词复现循环

- 标记词池中的单词，按艾宾浩斯间隔（1 天->2 天->4 天->7 天->15 天）自动排入每日复习队列
- 每次复现仍走 2 秒计时流程，连续 3 次正确则移出标记词池，状态变为“已掌握”
- 首页每天显示“今日待复习词：N 个”，优先级高于新词学习

#### 3.2.4 词汇量估算

- 每 7 天从全词库随机抽 50 词做一次无声测试（不显示释义，用户自评“认识/模糊/不认识”）
- 根据测试结果估算当前词汇量，绘制增长曲线

---

#### 3.2.5 单词判定策略（本地四级检查 + LLM fallback + 滥用熔断）

单词判定采用三层防护架构，优先本地、LLM 兜底、滥用自动收敛：

**第一层：本地四级检查（覆盖 95%+ 场景）**

| 级别 | 方法 | 说明 | 示例 |
|------|------|------|------|
| L1 | 直接包含 | 用户答案包含标准释义，或标准释义包含用户答案 | 标准"放弃"，用户"放弃" → true |
| L2 | 关键词匹配 | 标准释义按分号/逗号拆分为关键词，逐个与用户答案比对 | 标准"adj. 复杂的；难懂的"，关键词"复杂的"/"难懂的"，用户"复杂的" → true |
| L3 | 近义词映射 | 预置近义词词典（~25 组高频通用词组），关键词命中近义词即通过 | 标准"重要的"，用户"关键的" → true（近义词映射） |
| L4 | 编辑距离 | Levenshtein 相似度 ≥ 0.65，且串长 ≥ 2、≤ 20 字 | 标准"难以理解的"，用户"难以理解" → true（相似度 0.80） |

本地四级任一命中即返回正确，不再走后续流程。

**第二层：LLM 语义判断（仅限零字重叠边缘 case）**

- **触发条件**：本地四级检查全部失败，**且**用户答案与标准释义关键词之间零字重叠（无一字相同）
- **零字重叠判断**：用户答案中的每个字与所有标准关键词取交集，交集为空则触发 LLM
- **反例（不触发 LLM）**："难以理解的" ↔ "晦涩的难懂的" → 零字重叠，触发 LLM
- **正例（不触发 LLM，判 false）**："复杂的" ↔ "繁杂的" → 共享"杂"字，有重叠，说明用户在尝试但答偏了，不浪费 LLM 调用，直接判错
- LLM prompt 要求：temperature=0，仅返回 true/false，超时 5s 默认判 false

**第三层：自动熔断（防止 LLM 调用滥用）**

| 熔断机制 | 触发条件 | 动作 | 恢复条件 |
|----------|----------|------|----------|
| 单词级降级 | 同一单词（word_id）累计触发 LLM ≥ 3 次 | 将该单词标记为"跳过校验"，用户说任何内容均判正确 | 人工后台重置；或词库更新后自动清除 |
| 用户级熔断 | 同一用户连续 LLM 调用返回 false ≥ 5 次 | 暂停该用户当日 LLM fallback 权限，回退到本地判 false | 次日 0:00 自动恢复 |

**设计理由**：

- 零字重叠（如"晦涩"↔"鲜为人知"）才是真正需要语义判断的场景，任何有字重叠的 case 说明用户在回忆相关汉字，只是没回忆对，不应浪费 LLM 资源
- 单词级降级的假设：如果一个词的 3 种不同中文表达都无法被本地引擎覆盖，说明这个词的中文表达太多样，用语义相似度模型或 LLM 也无济于事，不如直接放行——背单词的核心目的是记忆，不是考试
- 用户级熔断的假设：连续多次 LLM 判 false 说明用户确实在瞎说，暂停 LLM 回退到本地严判即可，次日自动恢复，不搞封号

### 3.3 阅读精读模块（第二阶段核心）

#### 3.3.1 真题阅读库

- 收录 2000-2025 年英语一 + 英语二全部阅读真题（共约 200 篇）
- 每篇数据结构：{年份, Text编号, 原文(段落数组), 题目(5道, 每题含选项+正确答案+解析), 参考译文, 难度评级, 建议用时}

#### 3.3.2 阅读训练流程

**模式 A：限时做题模式**

- 计时器从建议用时开始倒数
- 5 道题在同一页面，支持上下滚动原文区域
- 提交后**不展示正确答案**，直接进入精读模式

**模式 B：逐句精读模式**

1. 原文按句拆分，每次展示一句
2. 用户手写/输入翻译后，可点击“查看参考译文”对比
3. **标记生词**：长按选中单词 -> 弹窗确认 -> 一键加入误解本
4. **标记长难句**：点击“标记为长难句”按钮，LLM 自动分析句子结构（主谓宾 + 从句类型），存入误解本
5. 全部句子过完后，**回到做题模式重新选答案**，此时再对答案
6. 对完答案后，每道错题弹出归因面板

#### 3.3.3 错题归因标注

归因选项：
- 单词不认识，关键词没看懂
- 长难句没读懂，理解偏差
- 逻辑理解偏差（段落主旨/作者态度判断错）
- 粗心/看错选项

归因数据存入用户模型，用于阅读模块的个人弱点分析和 LLM 阶段性诊断。解析由 LLM 生成，包含：定位原文、分析正确选项逻辑、解释干扰项排除原因。

#### 3.3.4 误解本（阅读模块部分）

- 每篇阅读完成后自动汇总：本篇标记的生词列表、标记的长难句列表（含 LLM 拆解结果）、错题列表（含归因标签 + 解析）
- 点击任意条目进入详情，支持复习和手动移除（表示已掌握）
- 进入误解本时展示鼓励文案：
  > *“精读一篇下来大概 60-70 分钟。2000-2009 年共约 40 篇阅读，40 天做完。精读 20 篇后你会发现质变。”*
  > *“你已经精读了 X 篇，还差 Y 篇达到质变点，继续加油。”*

### 3.4 作文训练模块（第三阶段核心）

#### 3.4.1 模板工厂

**大作文模板生成流程：**

1. 用户选择话题类型：**正面话题 / 负面话题 / 利弊分析 / 哲理名言**
2. LLM 基于话题类型生成一套 **半成品模板**，格式如下：

\`\`\`
第一段：描述图画
As is vividly depicted in the picture, [图画描述句]...
The message conveyed is thought-provoking: [主题提炼句]...

第二段：展开论述
There are several reasons accounting for this phenomenon.
First and foremost, [原因一]...
Moreover, [原因二]...
What’s more, [原因三]...

第三段：总结建议
From what has been discussed above, we may safely draw the conclusion
that [重申观点]... It is high time that [建议措施]...
\`\`\`

3. 模板中 `[...]` 部分为可替换槽位，用户需要了解每个槽位填什么
4. 用户可对生成的模板进行微调（改句式、增删段落），修改后保存为“我的模板”
5. 每种话题类型建议保存 1-2 套模板

**小作文模板生成流程：**

- 类型：建议信 / 投诉信 / 邀请信 / 感谢信 / 道歉信 / 通知
- 每种类型生成 1 套模板（约 8-10 句话），包含开头结尾固定句型 + 中间内容框架
- 模板生成后可进入练习模式

#### 3.4.2 真题作文练习

- 计时 30 分钟（小作文 15 分钟）
- 用户可选择自己保存的模板作为起点，在模板基础上填入主题相关内容
- 提交后 LLM 逐句批改

#### 3.4.3 LLM 批改与润色

提交后 LLM 返回结构化批改结果：

\`\`\`
=== 批改报告 ===

### 3.5 新题型训练

#### 3.5.1 题型覆盖

- 英语一：七选五 / 排序题 / 标题匹配
- 每篇限时 20 分钟

#### 3.5.2 交互设计

- 七选五：拖拽选项到空白处
- 排序题：拖拽段落调整顺序
- 标题匹配：点击段落 -> 选择对应标题
- 提交后立即出分，错题展示解析（由 LLM 生成解题逻辑：关键词/逻辑词/代词指代等线索）

#### 3.5.3 训练节奏

- 此模块优先级低于阅读和作文，在第三阶段随套卷穿插
- 不设独立精读流程，练完->对答案->看解析即可

---

### 3.6 翻译训练

#### 3.6.1 交互设计

- LLM 语义评分（0-10 分），非精确匹配
- 评分维度：语义完整性 / 流畅度 / 关键词准确度
- 展示参考译文，标注用户译文与参考译文的差异点
- 翻译模块定位为“低 ROI 随缘练”，每日最多 2 句，不做强制任务

---

### 3.7 套卷模拟

#### 3.7.1 全真模考

- 严格计时 3 小时，分 Section 展示
- 提交后自动算分，生成各题型得分报告
- 分数存入趋势数据

#### 3.7.2 模考分析

- 每次模考后 LLM 输出诊断：
  - 各题型得分 vs 目标分差距
  - 相比上次模考进步/退步的题型
  - 下阶段重点调整建议

### 3.8 数据看板

#### 3.8.1 首页仪表盘

首页展示三个关键指标卡片：词汇量、阅读正确率、预估分数。下方展示今日任务清单（根据所处阶段动态生成）和连续打卡天数。

#### 3.8.2 详细数据页

| 错题归因分布饼图 | 单词/语法/逻辑/粗心占比变化 |

#### 3.8.3 预计分数

- 基于最近 3 次模考各题型得分 x 权重，滚动计算预估总分
- 与目标分的差距用进度条展示

### 3.9 误解本（全局沉淀）

#### 3.9.1 数据来源

误解本是一个跨模块的“个人弱点数据库”，数据来自：

| 来源模块 | 沉淀内容 |
|----------|----------|
| 单词速记 | 标记词（多次未通过的单词） |
| 阅读精读 | 标记生词、标记长难句、错题（含归因标签+解析） |
| 翻译练习 | LLM 评分低于 5 分的句子 |
| 套卷模考 | 各题型错题 |

#### 3.9.2 误解本界面

- 分类筛选：单词 / 长难句 / 错题
- 单词类支持快速闪卡复习
- 用户标记“已掌握”后移入“已归档”，不再出现在活跃列表中

条目举例：
- 单词条目：展示英文、中文释义、累计标记次数、来源（如 2005 Text1），支持点击“已掌握”移除
- 长难句条目：展示原句、LLM 拆解的句子结构（主句 + with伴随状语 + that同位语从句），支持查看完整拆解
- 错题条目：展示题目、用户选择 vs 正确答案、归因标签、错误原因简述（如"assumption 理解为‘假设’而非‘承担’"），支持查看解析

### 3.10 进度教练（鼓励与微调系统）

#### 3.10.1 里程碑触发规则

- 每周日晚 20:00 推送一条周报

#### 3.10.3 节奏微调

- 用户可在“我的方案”页面点击“调整方案”

## 五、技术架构

### 5.1 总体架构

```
+-------------------------------------------+
|              微信小程序 (uni-app)            |
|  Vue 3 + TypeScript + uni-app 框架          |
|  状态管理: Pinia                             |
+--------------------+----------------------+
                     | HTTPS REST API
+--------------------v----------------------+
|            Java SpringBoot 后端             |
|  SpringBoot 3.x + MyBatis-Plus + WxJava    |
|  LLM 路由层: DeepSeek-V3 API               |
+--------------------+----------------------+
                     |
     +---------------+---------------+
     v               v               v
+---------+    +---------+    +----------+
|  MySQL  |    |  Redis  |    | 云存储    |
| 用户数据 |    | LLM缓存 |    | 词库JSON  |
| 学习记录 |    | 会话管理 |    | 真题原文  |
| 错题本  |    | 限流控制 |    | 静态资源  |
+---------+    +---------+    +----------+
```

### 5.2 前端：uni-app (Vue 3 + TypeScript)

| 能力 | 说明 |
|------|------|
| 跨端打包 | 微信小程序为主目标，uni-app 天然支持一键打包支付宝/字节/H5 |
| 状态管理 | Pinia（Vue 3 官方推荐，比 Vuex 轻量） |
| 网络请求 | `uni.request` + 封装拦截器（统一 token 注入、错误处理） |
| 本地存储 | `uni.setStorageSync` 缓存词库进度、阅读记录；离线时可用 |
| 语音能力 | `uni.getRecorderManager()` 录音 → SpringBoot 中转 → Python 调百度 ASR（免费 5 万次/天） |
| UI 框架 | uView Plus 3.x（uni-app 生态最成熟的 Vue 3 组件库） |

### 5.3 后端：Java SpringBoot 3.x

| 能力 | 技术选型 | 说明 |
|------|----------|------|
| 基础框架 | SpringBoot 3.x | Java 17+，长期支持 |
| ORM | MyBatis-Plus | 代码生成器 + 分页插件，适合数据密集型场景 |
| 微信集成 | **WxJava** (weixin-java-miniapp) | Java 生态最成熟的微信 SDK，覆盖登录、支付、订阅消息、内容安全等全部 API。GitHub 30k+ stars，持续维护 |
| LLM 调用 | OkHttp + DeepSeek SDK | DeepSeek 兼容 OpenAI 格式，也可直接用 OpenAI Java SDK 指向 DeepSeek endpoint |
| 缓存 | Spring Cache + Redis | 注解驱动缓存，LLM 结果按 key 自动缓存/失效 |
| 认证 | JWT | 微信登录后签发 JWT，小程序端存储 token |
| API 文档 | Knife4j (Swagger) | 自动生成接口文档，前后端联调 |
| 文件上传 | Spring MultipartFile | 接收小程序上传的图片/文件 -> 转存云存储 |
| 定时任务 | XXL-Job 或 Spring Scheduled | 每周日晚推送周报、词汇量定期测试提醒 |

**WxJava 接入评估：完全可行。** WxJava 对小程序的支持包括：
- `WxMaService`：登录（code2session）、用户信息解密
- `WxMaMessageService`：订阅消息推送（进度教练的推送能力）
- `WxMaQrcodeService`：小程序码生成
- 内容安全检测（`checkText`、`checkImage`）：用户作文内容审核

不需要自己封装微信 API，WxJava 已经做好了。

### 5.4 数据库：MySQL 8.0

选择 MySQL 而非 PostgreSQL 的理由：
- SpringBoot + MyBatis-Plus + MySQL 是 Java 生态最成熟的组合
- 学习记录表预估数据量在百万级以内，MySQL 完全胜任
- JSON 类型（MySQL 5.7+）可替代 PostgreSQL 的 JSONB，存储 LLM 返回的非结构化数据

**核心表设计（概要）：**

| 表名 | 职责 | 预估行数/用户 |
|------|------|:--:|
| `user` | 用户基础信息 + 7 题答案 + 专属方案 JSON | 1 |
| `word_progress` | 单词掌握状态（word_id + 标记次数 + 艾宾浩斯历史） | ~3600 |
| `word_mistake_pool` | 标记词池（待复习单词队列） | ~500 |
| `reading_record` | 每篇阅读的做题记录（答案 + 耗时 + 正确数） | ~100 |
| `reading_sentence` | 逐句精读记录（用户翻译 + 标记生词 + 标记长难句） | ~1000 |
| `mistake_book` | 误解本（跨模块聚合：单词/长难句/错题） | ~300 |
| `essay_record` | 作文练习记录（原文 + LLM 批改结果 JSON） | ~30 |
| `exam_record` | 模考记录（各题型得分 + LLM 诊断 JSON） | ~10 |
| `check_in` | 每日打卡记录 | ~180 |
| `weekly_report` | 周报历史 | ~24 |

### 5.5 缓存策略（Redis）

| 缓存内容 | Key 设计 | TTL | 说明 |
|----------|----------|-----|------|
| 阅读长难句拆解 | `reading:parse:{year}:{text}:{sent_idx}` | 永久 | 同一篇目所有用户共享 |
| 阅读错题解析 | `reading:analysis:{year}:{text}:{q_idx}` | 永久 | 同一道题解析不变 |
| 翻译参考评分 | `trans:score:{year}:{sent_hash}` | 永久 | 按句意 hash 缓存 |
| 用户会话 token | `session:{openid}` | 2h | JWT refresh 用 |
| LLM 调用限流 | `rate:{openid}:{action}` | 1min | 防止误操作刷 LLM |

### 5.6 语音识别

- 方案：百度 ASR 短语音识别（免费额度 5 万次/天）
  - 链路：`uni.getRecorderManager()` 前端录音(WAV) → `uni.uploadFile` → SpringBoot `SpeechController` → Python `stt.py` 子进程 → 百度 ASR API → 返回中文文本
  - SpringBoot 接收音频文件后保存为临时文件，通过 `ProcessBuilder` 调用 Python 脚本，脚本内使用 `urllib` 直接调百度 ASR HTTP API
  - 自动格式适配：SpeechService 检测文件头（非 RIFF/WAV 则调用 ffmpeg 转为 16kHz mono WAV），兼容模拟器 WebM、真机 WAV 等格式
  - Python 脚本位于 ackend/scripts/stt.py，无外部依赖（仅标准库）
  - Java ProcessBuilder 设置 PYTHONIOENCODING=utf-8，确保 Python 输出中文时编码正确
  - 缺点：链路多一跳（前端→后端→Python→百度），延迟约 2-5s；需本机安装 Python 3
  - 优点：百度中文识别效果稳定，免费额度充足；不依赖微信平台 API 可用性

### 5.7 词库数据方案

- **数据源**：GitHub 上现成的考研词汇 JSON 数据集（爬取获取）
- **必需字段**：英文单词、中文释义、词性、所属 Unit
- **可选字段**：音标、真题例句（能拿到就拿，拿不到不影响核心功能）
- **交付方式**：JSON 文件上传至云存储 -> 小程序首次启动下载到本地 -> 后续增量更新

## 六、风险与应对

| 风险 | 等级 | 应对方案 |
|------|:--:|----------|
| LLM 调用延迟（单词判断场景） | 高 | 四级本地检查覆盖 95%+ 场景；仅零字重叠边缘 case 调 LLM；LLM 具备自动熔断（单词级降级+用户级熔断）防止滥用 |
| LLM 生成方案不合理（幻觉） | 高 | Prompt 嵌入硬约束规则表，输出前校验：总分偏差 > 3 分则拒绝并重新生成 |
| 词库数据构建工作量大 | 中 | 爬取 GitHub 现成 JSON 数据集，非必需字段（音标/例句）不强求 |
| 阅读逐句拆分不准确 | 中 | spaCy 英文分句 + 人工抽检；注意处理 Mr./U.S./引号等边界 case |
| 小程序包大小超限 | 中 | 词库和真题原文放云存储，按需下载缓存；仅 UI 代码留在小程序包内 |
| 百度 ASR 免费额度耗尽或服务异常 | 低 | 本地降级：倒计时超时或"不认识"按钮仍可完成单词闪卡核心流程；百度免费额度 5 万次/天，单人日均调用约 200 次，充足 |
| LLM 批改一致性 | — | 用户决策：接受。实际使用中用户大概率不会在小程序内写完整大作文，批改场景有限 |
| 版权 | — | 用户决策：不考虑。产品仅面向熟人使用，不上线公开 |

## 七、非功能需求

### 5.1 性能

- 单词闪卡切换延迟 < 200ms
- LLM 批改作文响应时间 < 10s（需 loading 动画）
- 语音识别 + LLM 判断单词释义总延迟 < 3s

### 5.2 离线能力

- 单词闪卡（纯本地模式）支持离线使用
- 阅读原文可缓存到本地，离线可查看
- LLM 相关功能（批改、拆解、评分）需联网

### 5.3 数据安全

- 用户学习数据本地存储 + 云端同步（微信云开发）
- 作文内容仅用于批改，不做他用（隐私协议明确）

### 5.4 兼容性

- 微信小程序最低版本要求：微信 8.0+
- iOS 14+ / Android 8+
