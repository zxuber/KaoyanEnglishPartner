## 开发总览

本项目分为三期交付，每期约 4-6 周。开发顺序遵循「能跑通核心闭环 → 能验证用户价值 → 功能完整」的原则。

```
第一期（MVP）        第二期（核心闭环）      第三期（完整产品）
   4-6 周               6-8 周                 4-6 周
   ├─ M1 学习画像       ├─ M6 阅读做题         ├─ M10 作文模板
   ├─ M2 方案生成       ├─ M7 阅读精读         ├─ M11 作文批改
   ├─ M3 单词闪卡       ├─ M8 错题+误解本      ├─ M12 套卷模考
   ├─ M4 数据看板       ├─ M9 LLM 接入         ├─ M13 新题型
   └─ M5 用户认证                              ├─ M14 翻译
                                               ├─ M15 进度教练
                                               └─ M16 词汇测试
```

## 第一期：MVP 基础闭环（4-6 周）

### M1：学习画像（分模块采集页面）

| 维度 | 说明 |
|------|------|
| **目标** | 用户首次进入小程序，完成“学习画像”采集，数据足以支撑诊断摘要与可执行计划生成，并据此进入首页首个训练任务 |
| **负责** | `frontend-dev` (页面) + `backend-dev` (API) |
| **前端任务** | 1. 保持现有“学习画像”题目结构稳定，不额外扩题<br>2. 保持一屏一题交互与诊断摘要确认页<br>3. 方案生成后展示更细颗粒度的计划卡片<br>4. 完成后直接落到“今日训练首页”，不额外插入模块选择页 |
| **后端任务** | 1. 扩展 `POST /api/v1/users/onboarding` 入参结构<br>2. 新增诊断摘要生成逻辑<br>3. 调整 prompt，使方案输出包含诊断、周计划、日模板、风险补救<br>4. `GET /api/v1/users/{id}/plan` 返回更完整 JSON |
| **测试任务** | 1. 新增字段参数校验测试<br>2. 诊断摘要与完整计划链路测试<br>3. 方案生成稳定性测试（连续 10 次生成，检查结构完整性而非只看总分偏差） |
| **验收标准** | 用户完成学习画像 → 先看到诊断摘要 → 再生成完整计划 → 直接进入今日训练首页 → 输出包含阶段/周/日/风险四层结构 |
| **预估工时** | 前端 4d + 后端 3d + 测试 1.5d = 8.5d |

### M2：方案生成（LLM 专属计划）

| 维度 | 说明 |
|------|------|
| **目标** | LLM 生成结构化专属学习方案，含分数拆解、阶段节奏、每日任务、里程碑 |
| **负责** | `backend-dev` |
| **核心任务** | 1. 编写 LLM prompt 模板，嵌入硬约束规则表<br>2. 实现输出 JSON 解析 + 校验（总分偏差、阶段天数合理性）<br>3. 不合法输出自动重试（最多 3 次）<br>4. 方案存入 `user.plan_json` 字段 |
| **测试任务** | 1. 覆盖 5 种典型用户画像（基础好/中/差 × 时间多/少）<br>2. 边界测试：极端输入（目标分 40 / 80，剩余 30 天 / 300 天） |
| **验收标准** | 所有测试画像的方案偏差 ≤ 3 分，阶段天数与剩余天数一致 |
| **预估工时** | 后端 3d + 测试 2d = 5d |

> 当前状态：已接通首版，学习画像提交会真实调用 DeepSeek 并返回结构化方案。

### M3：单词闪卡（本地判断版本）

| 维度 | 说明 |
|------|------|
| **目标** | 对话式单词闪卡，2 秒倒计时 + 语音输入 + 本地语义判断，不依赖 LLM |
| **负责** | `frontend-dev` (页面+语音交互) + `backend-dev` (判断逻辑+词库API) |
| **前端任务** | 1. 单词卡片组件：英文+音标展示，中文遮罩隐藏<br>2. 2 秒倒计时进度条（绿→黄→红）<br>3. 按住录音按钮，调用 uni.getRecorderManager() 录音(WAV) → uni.uploadFile 上传后端 → 后端调 Python 脚本 → 百度 ASR 识别 → 返回中文文本<br>4. 识别结果发送后端判断，展示正确/错误动画<br>5. 每轮 20 词，结束后展示本轮正确率<br>6. 「不认识」按钮，展示释义 3 秒后跳过 |
| **后端任务** | 1. 词库导入（爬取 GitHub JSON → 解析 → 存入 `word` 表）<br>2. 本地判断引擎：近义词词典 + Levenshtein 编辑距离 + 词向量相似度<br>3. `POST /api/v1/words/check` 判断接口（超时 1s）<br>4. `POST /api/v1/words/progress` 更新掌握状态<br>5. `GET /api/v1/words/review` 获取今日待复习词（艾宾浩斯算法） |
| **测试任务** | 1. 本地判断准确率测试（人工标注 500 组「用户说→标准释义」对照集）<br>2. 艾宾浩斯复现循环正确性<br>3. 离线模式：无网络时闪卡仍可用 |
| **验收标准** | 本地判断准确率 ≥ 85%（对照集），闪卡切换 < 200ms，语音→判断→结果 < 3s |
| **预估工时** | 前端 5d + 后端 4d + 测试 3d = 12d |

### M4：今日训练首页骨架

| 维度 | 说明 |
|------|------|
| **目标** | 首页固定承载最高频的今日单词训练入口，再用次级卡承接继续训练，并把推荐理由合并到专项入口 |
| **负责** | `frontend-dev` (页面) + `backend-dev` (统计API) |
| **前端任务** | 1. 首页最大主卡固定为 `今日单词训练`，入口固定到背单词页<br>2. 主卡下方放较小的 `继续上次训练` 卡片，展示 recent training 进度<br>3. 删除独立 `今日推荐` 区块，将推荐 subtitle / reason / badge 合并进专项入口卡片<br>4. 关键指标卡片与趋势图下沉为辅助信息区，不抢占首屏<br>5. 保留打卡入口，但不作为首页主驱动 |
| **后端任务** | 1. `GET /api/v1/users/{id}/dashboard` 聚合统计数据、最近训练状态与画像驱动的推荐任务<br>2. 推荐任务不再单独渲染为首页区块，而是用于增强专项入口卡片<br>3. `POST /api/v1/users/{id}/checkin` 打卡接口<br>4. 连续打卡天数计算 |
| **测试任务** | 1. 统计数据准确性校验<br>2. 打卡去重（同一天多次打卡只计一次） |
| **验收标准** | 首页加载 < 1s，用户首屏能明确进入今日单词训练；继续上次训练不抢主入口；专项训练能展示画像驱动的 `今日优先` 标签与推荐理由 |
| **预估工时** | 前端 3d + 后端 2d + 测试 1d = 6d |

## 第二期：核心闭环（6-8 周）

### M6：阅读做题模式

| 维度 | 说明 |
|------|------|
| **目标** | 真题阅读限时做题，原文滚动+题目下方选择，计时器倒计时 |
| **负责** | `frontend-dev` (页面) + `backend-dev` (真题数据API) |
| **前端任务** | 1. 阅读页面：原文区域可滚动 + 5 道题固定在下方<br>2. 计时器从建议用时倒数，超时红色闪烁提醒<br>3. 选项点击选中高亮，支持修改<br>4. 提交后不展示正确答案，直接进入精读模式 |
| **后端任务** | 1. 真题数据导入（2000-2025 年阅读真题）<br>2. `GET /api/v1/readings?year=&text=` 获取阅读原文+题目<br>3. `POST /api/v1/readings/{id}/submit` 提交答案（仅记录，不返回正误） |
| **测试任务** | 1. 原文滚动性能（长文本不卡顿）<br>2. 计时器准确性<br>3. 答案提交后不暴露正确答案 |
| **验收标准** | 原文渲染 < 1s，滚动流畅 60fps，计时器误差 < 1s |
| **预估工时** | 前端 4d + 后端 3d + 测试 2d = 9d |

### M7：阅读精读模式

| 维度 | 说明 |
|------|------|
| **目标** | 逐句展示原文，用户输入翻译，标记生词/长难句 |
| **负责** | `frontend-dev` (精读页面) + `backend-dev` (分句API+标记API) |
| **前端任务** | 1. 原文按句拆分展示，每次一句<br>2. 用户输入翻译区域<br>3. 长按单词选中 → 弹出「加入误解本」确认<br>4. 「标记长难句」按钮 → 调用 LLM 拆解<br>5. 「查看参考译文」展开/收起<br>6. 全部句子过完后，回到做题模式重新选答案 |
| **后端任务** | 1. `GET /api/v1/readings/{id}/sentences` 分句列表+参考译文<br>2. `POST /api/v1/readings/sentences/{id}/mark-word` 标记生词<br>3. `POST /api/v1/readings/sentences/{id}/mark-difficult` 标记长难句 → 触发 LLM 拆解<br>4. `GET /api/v1/readings/{id}/compare` 对比用户两次答案变化 |
| **验收标准** | 逐句切换 < 200ms，长难句 LLM 拆解 < 5s |
| **预估工时** | 前端 5d + 后端 3d + 测试 2d = 10d |

### M8：错题归因 + 误解本

| 维度 | 说明 |
|------|------|
| **目标** | 错题四选一归因 + 跨模块误解本聚合 |
| **负责** | `frontend-dev` (归因面板+误解本页面) + `backend-dev` (聚合API) |
| **前端任务** | 1. 对答案后，每道错题弹出归因面板（单词/长难句/逻辑/粗心 四选一）<br>2. 误解本页面：分类筛选（单词/长难句/错题）<br>3. 单词条目：展示英文+释义+累计标记次数+来源<br>4. 长难句条目：展示原句+LLM 拆解结果<br>5. 错题条目：展示题目+用户答案+正确答案+归因标签+解析<br>6. 「已掌握」按钮，移入已归档 |
| **后端任务** | 1. `POST /api/v1/mistakes` 记录错题+归因标签<br>2. `GET /api/v1/mistakes?type=word|sentence|question` 分类查询<br>3. `PATCH /api/v1/mistakes/{id}/archive` 归档<br>4. `GET /api/v1/mistakes/stats` 归因分布统计（饼图数据） |
| **验收标准** | 误解本加载 < 1s，分类筛选正确，归档后不再出现在活跃列表 |
| **预估工时** | 前端 4d + 后端 2d + 测试 2d = 8d |

> 3 tab 分支首版已先落地“轻资产库”方向：底部单独保留 `误解本` tab，现有 `单词/短句` 继续走用户动态误解本；新增 `写作表达/固定搭配/易混词` 走全局资产库 `mistake_asset_library` + 用户进度表 `mistake_asset_progress`。初始化脚本已提供三类各 100 条原始数据，且同一分类下 `source_text` 唯一；误解本页新增 `DONE` 模块，但只收录误解本中已点击 `done` 的单词，作为“已掌握词库”回看，不混入短句和系统资产库。后续可逐步支持用户主动收藏、按来源筛选和跨题型自动沉淀。

### M9：LLM 接入（长难句拆解 + 错题解析）

| 维度 | 说明 |
|------|------|
| **目标** | 后端统一 LLM 路由层，支持 DeepSeek-V3 调用 + Redis 缓存 + 超时降级 |
| **负责** | `backend-dev` |
| **核心任务** | 1. LLM 路由层：根据任务类型选择 prompt 模板<br>2. Redis 缓存：同篇同句拆解只调一次<br>3. 超时降级：> 10s 返回预设降级响应<br>4. 错题解析生成：定位原文 + 正确选项逻辑 + 干扰项排除原因 |
| **测试任务** | 1. 缓存命中测试（二次请求不走 LLM）<br>2. 超时降级测试（模拟网络延迟）<br>3. Prompt 稳定性测试（相同输入多次调用结果一致性） |
| **验收标准** | 缓存命中时响应 < 100ms，LLM 调用超时自动降级不报错 |
| **预估工时** | 后端 4d + 测试 2d = 6d |

### M10-lite：写作搭建训练 MVP（已增强）

| 维度 | 说明 |
|------|------|
| **目标** | 先解决用户“不会开头、不会展开、不会结尾”，但不能停留在口头思路训练；MVP 应让用户在高质量框架中完成局部自由写作 |
| **负责** | `frontend-dev` + `backend-dev` |
| **产品流程** | 1. 写作二级页选择 `小作文 / 大作文`<br>2. 系统给一题仿考研风格写作任务<br>3. 采用卡片式 Step Flow，同页分步不跳页<br>4. 用户逐题选择 `开头起笔 / 中间展开 / 结尾收束`，每次只展示一个方向问题<br>5. 用户单独进入“大白话说想法/卡点”卡片<br>6. AI 先分析用户选项是否有偏差，再判断用户补充内容是 `真实写作思路` 还是 `求助/焦虑/发牢骚`<br>7. 如果用户不会写，AI 应先鼓励并降级任务，不应假装用户已经完成作文思路<br>8. 系统给出考研范文水准的半填空作文框架，框架负责高级句式、语气和连接<br>9. 用户逐空完成局部自由写作，每次只展示一个空白，可写短语、一个句子或 1-3 句<br>10. AI 针对用户填入内容批改：贴题度、逻辑、语法、表达升级<br>11. 输出可沉淀的写作表达 / 固定搭配 |
| **后端接口** | `GET /api/v1/practice/writing/task?userId=&type=small|large`<br>`POST /api/v1/practice/writing/coach` |
| **验收标准** | 用户不用面对整篇空白作文，也能完成一次“选方向 -> 高质量框架 -> 局部自由写作 -> AI 批改 -> 资产沉淀”的训练闭环 |
| **质量要求** | 半填空框架不能是小学生/初中生表达；应使用考研可用但不过度生僻的句式。用户填入内容可以简单，整体作文质感由系统框架托底 |
| **后续演进** | 增加题库表、写作框架表、用户填空记录表、整篇作文批改、按英语一/英语二自动匹配题型 |

### M14-lite：翻译专项 MVP（已落地）

| 维度 | 说明 |
|------|------|
| **目标** | 把翻译训练拆成稳定动作：找主干、拆修饰、顺中文 |
| **负责** | `frontend-dev` + `backend-dev` |

| **后端接口** | `GET /api/v1/practice/translation/task?userId=`<br>`POST /api/v1/practice/translation/coach` |

### M15-lite：完形填空 MVP（已落地）

| 维度 | 说明 |
|------|------|
| **目标** | 先用单空训练验证“选择答案 + 说理由 + AI 追问”的完形教练体验 |
| **负责** | `frontend-dev` + `backend-dev` |
